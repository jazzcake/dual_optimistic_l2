## **이중 낙관적 파이프라인: DAG 합의와 병렬 실행 엔진의 결합 아키텍처**

### **초록 (Abstract)**

기존의 모듈러 블록체인 시스템은 합의 레이어에서 트랜잭션의 글로벌 순서가 완전히 확정된 이후에만 실행 레이어가 연산을 시작하는 직렬화된 파이프라인을 갖습니다. 이는 합의 대기 시간 동안 실행 엔진이 유휴 상태에 머물게 하여 전체 네트워크의 처리량을 제한하는 근본적인 병목이 됩니다.

본 문서는 미스티세티(Mysticeti) 아키텍처로 대표되는 DAG 기반 합의 알고리즘의 초저지연 3델타 최종성(3-Delta Finality) 구조와 다중 버전 상태 기반의 병렬 실행 엔진을 결합한 이중 낙관적 실행 파이프라인을 제안합니다. 합의 완료 전 로컬 쿼럼이 형성되는 2델타 시점의 시간적 가속과, 트랜잭션 충돌을 제어하는 공간적 가속을 수직으로 통합하여 상태 롤백 오버헤드 없이 시스템의 체감 성능을 극대화하는 구조를 명세합니다.

### **핵심 시스템 컴포넌트**

1. DAG 기반 초저지연 합의 모듈 (DAG-based Low-Latency Consensus Module)
트랜잭션의 인과적 역사를 수집하여 전파하는 레이어로, 미스티세티 모델의 3단계 전파 위상을 활용해 두 개의 결정적 트리거를 분리하여 발생시킵니다.
    - 1단계 (1델타, 제안): 각 노드가 트랜잭션을 담은 블록을 네트워크에 브로드캐스트합니다.
    - 2단계 (2델타, 인증 및 소프트 커밋): 다른 노드들로부터 2f+1 이상의 참조(Reference)를 받아 로컬 뷰에서 쿼럼이 형성되는 시점입니다. 합의 모듈은 글로벌 순서가 완전히 보장되지는 않았으나 확률적으로 매우 안전한 이 시점에 실행 레이어로 소프트 커밋 신호를 보냅니다.
    - 3단계 (3델타, 확정 및 하드 커밋): 라운드 리더가 결정되고 글로벌 위상이 정렬되어 블록의 최종 순서가 불가역적으로 확정되는 시점입니다. 합의 모듈은 이때 하드 커밋 신호를 발생시킵니다.
2. 낙관적 파이프라인 스케줄러 (Optimistic Pipeline Scheduler)

    합의 모듈과 실행 모듈 사이의 비동기 신호 중재자 역할을 수행합니다. 합의 모듈로부터 2델타 소프트 커밋 신호를 수신하면 해당 블록을 즉시 실행 큐에 적재합니다. 동시에 라운드 인덱스 기반으로 대기열을 엄격하게 관리하여, 네트워크 지연으로 인한 패킷 순서의 역전이 발생하더라도 로컬 노드 간의 전체 순서 정렬을 수학적으로 보장합니다.

3. 상태 추상화 기반 병렬 실행 모듈 (State Abstraction-based Parallel Execution Module)

    스케줄러가 전달한 2델타 블록 내의 트랜잭션들을 메인 데이터베이스(DB)가 아닌, 완전히 격리된 다중 버전 섀도우 메모리(Multi-Version Shadow Memory) 상에서 병렬로 실행합니다. 트랜잭션 간의 읽기/쓰기 충돌을 메모리 레벨에서 감지하고 스케줄링함으로써 연산 시간을 획기적으로 단축합니다.

4. 롤백 및 확정 래퍼 모듈 (Rollback and Commit Wrapper Module)

    합의 모듈로부터 3델타 하드 커밋 신호를 수신했을 때 최종 상태를 판별합니다. 3델타의 글로벌 합의 결과가 2델타 시점의 로컬 판단과 일치하면, 섀도우 메모리의 상태 변화량(State Diff)을 메인 원장에 영구적으로 병합합니다. 만약 불일치가 발생할 경우, 개별 트랜잭션을 추적하지 않고 해당 라운드의 섀도우 메모리 전체를 즉시 폐기(Drop)하여 분산 시스템 특유의 무거운 롤백 오버헤드를 제거합니다.


### **성능 평가 및 수학적 모델링 (Performance Evaluation)**

본 장에서는 허가형 환경을 가정하여 롤백이 발생할 확률이 $0$에 수렴한다고 전제하고, 글로벌 확정 후 실행하는 기존 모델(3델타 대기)과 본 제안 모델의 지연 시간 및 처리량 이득을 수학적으로 분석합니다.
변수 정의

- $\Delta$: 네트워크 노드 간 메시지가 1홉(hop) 전파되는 데 걸리는 평균 지연 시간.
- $E$: 병렬 EVM이 1개 라운드의 앵커 블록에 담긴 트랜잭션 배치를 실행하는 데 걸리는 총 연산 시간.

**기준 모델 (3델타 확정 후 직렬 실행)**

- 체감 최종 완료 시간: $3\Delta + E$.

**제안 모델 (2델타 낙관적 병렬 실행)**

- 로컬 실행 시작 시점: $2\Delta$.
- 글로벌 합의 완료 시점: $3\Delta$.
- 체감 최종 완료 시간: $\max(3\Delta, 2\Delta + E)$.

**지연 시간 단축 이득 (Latency Gain)**

두 아키텍처의 체감 지연 시간 차이는 $(3\Delta + E) - \max(3\Delta, 2\Delta + E)$ 로 도출되며, 이는 시스템 환경에 관계없이 항상 $\min(\Delta, E)$ 만큼의 절대적인 시간 단축을 보장합니다.

- 연산이 매우 빠른 경우 ($E \le \Delta$): 실행 시간이 3델타 대기 시간 내에 완전히 은닉되어 시스템 체감 지연 시간은 순수 합의 속도인 $3\Delta$로 수렴합니다.
- 연산이 느린 경우 ($E > \Delta$): 기존 모델에서 유휴 상태로 버려지던 $1\Delta$의 시간을 선행 연산에 활용하여, 정확히 $\Delta$ 만큼의 지연 시간을 확정적으로 절약합니다.

**자원 활용률 및 처리량 (Throughput) 극대화**

기존 실행 모델에서는 매 블록마다 글로벌 합의를 기다리는 $1\Delta$ 시간 동안 CPU 파이프라인이 유휴 상태에 빠집니다. 제안 아키텍처는 이 틈새 구간을 다음 블록의 연산으로 선제적으로 채워 실행 엔진의 겹침(Overlapping) 구간을 생성합니다. 이는 복잡한 컨트랙트가 많은 L2 환경에서 하드웨어 병목을 해소하고 시스템 전체의 최대 처리량(Max TPS)을 선형적으로 증가시키는 핵심 동력으로 작용합니다.

### 트랜잭션 충돌 저항성 및 배압 제어 (Transaction Conflict Resilience & Backpressure)

본 아키텍처는 합의 레이어의 거시적 낙관과 실행 레이어의 미시적 낙관을 수직으로 결합한 구조입니다. 실행 레이어 내부에서는 스마트 컨트랙트 상태 충돌로 인해 빈번한 롤백이 발생할 수 있으나, 시스템 전체의 퍼포먼스를 보호하기 위해 다음과 같은 방어 기제를 갖춥니다.

**매크로 롤백과 마이크로 롤백의 완벽한 물리적 격리**

스케줄러가 2델타 시점에 병렬 EVM으로 전달하는 것은 상태가 아닌 트랜잭션의 순서입니다. EVM 내부에서 트랜잭션 간의 읽기 및 쓰기 충돌이 발생하여 재실행되더라도, 이는 완전히 격리된 섀도우 메모리 내부에서 포인터만 다시 조정하는 마이크로 롤백에 불과합니다. 이러한 실행 레이어의 내부 충돌은 합의 레이어의 초고속 3델타 글로벌 확정 프로세스에 어떠한 간섭도 일으키지 않으며, 두 파이프라인의 물리적 독립성을 완벽하게 보장합니다.

**수학적 퍼포먼스 방어 (상대적 우위 유지)**

트랜잭션 충돌로 인한 재실행이 전체 시스템의 지연 시간에 미치는 영향을 수학적으로 증명하면 다음과 같습니다. 충돌 없이 처리된 이상적인 연산 시간을 $E_{base}$, 충돌로 인해 낭비된 재실행 시간을 $E_{retry}$라고 정의할 때, 총 연산 시간은 $E = E_{base} + E_{retry}$가 됩니다.

- 기준 모델 (3델타 확정 후 직렬 실행)의 체감 완료 시간: $3\Delta + E_{base} + E_{retry}$
- 제안 모델 (2델타 낙관적 병렬 실행)의 체감 완료 시간: $\max(3\Delta, 2\Delta + E_{base} + E_{retry})$

위 수식은 EVM 내부의 충돌률이 극단적으로 치솟아 $E_{retry}$가 폭증하더라도, 제안 모델이 직렬 실행 모델 대비 선제적으로 확보한 최소 $1\Delta$ 분량의 시간적 이득은 절대 삭감되지 않음을 증명합니다. 실행 레이어에서 아무리 최악의 충돌 상황이 발생하더라도 수학적인 상대적 퍼포먼스 우위는 무조건 유지됩니다.

**유일한 병목 시나리오: 큐 밀림과 배압 제어 (Backpressure)**

마이크로 롤백의 파급력은 실행 레이어 내부로 제한되지만, 특정 블록에서 $E_{retry}$가 기하급수적으로 증가하여 총 연산 시간 $E$가 합의 주기를 크게 초과할 경우 시스템 전체의 처리량 한계에 부딪히게 됩니다. 합의 엔진은 지속적으로 다음 블록을 스케줄러에 밀어 넣는데 EVM의 처리가 지연되면, 실행 대기열이 무한정 적체되어 L1으로 제출해야 할 최종 체크포인트 생성이 멈추는 현상이 발생합니다.

이를 원천적으로 방지하기 위해 낙관적 파이프라인 스케줄러 내부에 배압 제어 로직을 필수적으로 탑재합니다. 실행 큐의 적체량이 시스템이 허용하는 임계치를 초과하게 되면, 스케줄러가 합의 엔진 측에 블록 제안 속도를 일시적으로 늦추라는 억제 신호를 피드백 루프로 발송하여 파이프라인의 붕괴를 사전에 차단합니다.

### BFT 합의 알고리즘 간의 낙관적 실행 결합 비교 분석 (DAG vs HotStuff)

본 아키텍처의 우수성을 입증하기 위해, 현재 최상위 성능을 내는 단일 리더 기반의 HotStuff 파생 합의 알고리즘(AptosBFT, MonadBFT 등)과 낙관적 병렬 EVM을 결합했을 때의 시나리오를 본 제안 모델(DAG 기반)과 1대1로 비교 분석합니다. 두 모델 모두 동일한 $3f+1$ 비잔틴 내성 환경을 가정합니다.

단일 리더의 매크로 롤백(네트워크 페널티) vs 다중 제안자의 상태 보존(CPU 페널티)

HotStuff 환경에서 $2\Delta$ (Pre-Commit) 시점에 낙관적 실행을 시작할 경우, 가장 큰 위험은 뷰 체인지(View Change)입니다. 해당 라운드의 리더가 비잔틴 노드이거나 네트워크 타임아웃이 발생하면 합의 자체가 무효화됩니다. 이는 실행 레이어가 섀도우 메모리에 선제적으로 처리해 둔 연산 결과를 100% 폐기해야 하는 매크로 롤백으로 이어지며, 새로운 리더가 블록을 전파할 때까지 파이프라인 전체가 정지하는 값비싼 네트워크 지연 페널티를 발생시킵니다.

반면 미스티세티 모델의 DAG는 모든 노드가 동시에 블록을 제안하는 다중 제안자 구조입니다. $f$개의 비잔틴 노드가 존재하더라도 로컬 뷰에서 $2f+1$ 쿼럼이 형성된 $2\Delta$ 시점의 데이터는 DAG 그래프 안에 온전히 보존됩니다. 3델타 시점의 글로벌 앵커가 비잔틴 노드여서 라운드가 스킵되더라도 데이터가 유실되지 않으며, 단지 글로벌 총정렬(Total Ordering) 순서만 다음 라운드에서 일부 재조정될 뿐입니다.

**순서 역전으로 인한 트랜잭션 충돌과 우아한 실패 (Graceful Degradation)**

DAG 환경에서 순서가 재조정되면 실행 레이어에서는 필연적으로 트랜잭션 단위의 읽기 및 쓰기 충돌이 발생합니다. 앞선 트랜잭션의 순서가 바뀌어 섀도우 메모리의 상태값이 오염되므로, 해당 트랜잭션과 의존성 그래프 하위의 노드들을 재연산해야 합니다. 그러나 이는 파이프라인이 멈추는 매크로 롤백이 아니라, 합의 엔진은 멈춤 없이 굴러가는 상태에서 실행 엔진의 CPU 스레드만 사용하여 재연산을 수행하는 미시적이고 격리된 실패에 불과합니다. 재연산 시간인 $E_{retry}$가 증가하더라도, 이미 $1\Delta$의 대기 시간을 활용해 돌려둔 선행 연산의 수학적 이득은 훼손되지 않습니다.

**허가형(Permissioned) 환경에서의 실질적 지연 제로화**

가장 중요한 점은 본 아키텍처가 허가형 노드 기반의 메인넷 환경에서 운영된다는 것입니다. 퍼블릭 체인과 달리 신원이 검증되고 통제된 밸리데이터들로만 네트워크가 구성되므로, 악의적인 이중 제안이나 고의적인 타임아웃을 유발하는 비잔틴 노드의 존재 확률은 실질적으로 0에 수렴합니다.

따라서 이론적으로 대비해 둔 앵커 스킵에 의한 대규모 순서 역전이나 그로 인한 $E_{retry}$의 급증 현상은 실제 운영 환경에서는 거의 발생하지 않습니다. 결과적으로 본 이중 낙관적 파이프라인은 엣지 케이스에 대한 완벽한 수학적 방어력을 갖추고 있으면서도, 실무적으로는 이론상 최대의 처리량과 최소의 지연 시간을 어떠한 네트워크 중단 없이 지속적으로 뽑아낼 수 있는 최적의 아키텍처로 기능합니다.
