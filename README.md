## 1. One-Line Thesis

합의와 실행 양쪽에서 동시에 optimistic하게 동작하는, 허가형 고성능 EVM Layer-2.

---

## 2. Problem

EVM 호환 체인들은 두 가지 병목에 직면한다.

**합의 병목**: 블록이 확정(finalize)될 때까지 실행을 시작하지 않으므로, 합의 지연이 곧 사용자 체감 지연이 된다. 전형적인 BFT 프로토콜(3-phase commit)에서는 3라운드(3δ)를 기다려야 실행이 가능하다.

**실행 병목**: EVM은 트랜잭션을 순차(sequential)로 처리한다. 상태 충돌이 없는 트랜잭션도 앞선 트랜잭션이 끝날 때까지 대기하므로, 하드웨어 병렬성을 활용하지 못한다.

기존 프로젝트들은 둘 중 하나만 공략한다. Monad는 실행 병렬화에 집중하고, MegaETH는 시퀀서 실행 최적화에 집중한다. 본 프로젝트는 양쪽을 동시에 공략한다.

**기존 고성능 합의 체인보다 빠른 이유**: Monad(MonadBFT), Aptos(AptosBFT) 등 현재 최상위 성능의 합의 알고리즘은 HotStuff 파생의 단일 리더 기반 BFT이다. 이들도 2δ 시점에 낙관적 실행을 시도할 수 있으나, 리더가 비잔틴이거나 타임아웃이 발생하면 뷰 체인지(View Change)로 합의 자체가 무효화되어 실행 결과를 100% 폐기해야 하는 **매크로 롤백**(파이프라인 정지 + 네트워크 지연 페널티)이 발생한다. 반면 DAG 기반 합의(Mysticeti)는 모든 노드가 동시에 블록을 제안하므로, 비잔틴 노드가 존재하더라도 2δ 쿼럼 데이터가 DAG에 보존된다. 순서 재조정은 발생할 수 있으나 데이터 유실이 없으며, 재연산은 CPU 스레드만 사용하는 **마이크로 롤백**에 불과하다. 이것이 DAG + Optimistic Execution 결합이 HotStuff + Optimistic Execution 결합보다 구조적으로 우월한 이유이다.

## 3. Architecture: Dual Optimistic

핵심 설계 결정은 하나다. **"뒤집힐 확률이 극히 낮은 시점에 미리 실행하고, 실행 자체도 충돌이 없다고 가정하고 병렬로 돌린다."** 양쪽 모두 잘못되면 되돌린다(optimistic). 복구 경로가 명확하므로 안전하다.

### 3.1 Optimistic Consensus — Mysticeti 2δ Execution

합의 엔진으로 SUI의 Mysticeti를 채택하되, **DAG 기반 합의(순서 결정)만 사용**한다. SUI 특유의 오브젝트 중심 데이터 구조는 사용하지 않는다. Mysticeti는 본 프로젝트에서 순수한 트랜잭션 순서 결정 엔진(ordering engine)으로 기능하며, 합의 출력은 "이 순서로 트랜잭션을 실행하라"는 정렬된 리스트이다.

Mysticeti는 DAG 기반 BFT로, 리더 병목 없이 모든 밸리데이터가 블록을 병렬 제안하며, 3δ(3라운드)에서 최종화된다. 본 프로젝트는 3δ 최종화를 기다리지 않고, **2δ 시점에 실행을 개시**한다.

**왜 2δ가 안전한가**: 2δ 시점에서 해당 블록은 이미 quorum(2/3+)의 투표를 확보한 상태다. 3δ에서 뒤집히려면 1/3 이상의 밸리데이터가 비잔틴이어야 하며, 이는 BFT의 기본 안전 가정(f < n/3) 위반에 해당한다. 허가형 네트워크에서 밸리데이터를 통제할 수 있으므로, 이 가정은 더욱 강하게 유지된다.

**핵심 게이트 구조**: "합의 → 실행 → 원장"은 세 단계로 분리되며, 명확한 게이트가 존재한다.

- **2δ 게이트 (실행 시작)**: 합의가 순서를 잠정 확정하면, 해당 순서대로 트랜잭션을 실행 파이프라인에 밀어넣는다.
- **3δ 게이트 (원장 커밋)**: 합의가 최종 확정되면, 실행 결과를 원장에 내린다. **3δ 신호를 받아야만 원장에 반영된다.**

**비잔틴 equivocation 대응**: 극단적으로 2δ에서 잠정 확정된 순서가 3δ에서 뒤집히는 경우(비잔틴 equivocation), 실행 파이프라인에 있는 트랜잭션 뭉치들의 순서를 재정렬하면 된다. 실행 결과는 아직 원장에 반영되지 않은 상태이므로, 상태 트리 롤백이 아니라 **실행 큐 재정렬**로 처리된다. 비용이 극히 낮다.

**효과**: 합의 최종화 대기 시간에서 1δ(1라운드)를 절약한다. Mysticeti의 일반적 라운드 시간이 수백 밀리초일 때, 이 절약은 사용자 체감 지연에 직접적인 영향을 미친다.

### 3.2 Optimistic Execution — REVM + Block-STM 병렬 실행 (섀도우 메모리)

실행 엔진은 REVM(Rust EVM)을 기반으로 하며, Aptos의 Block-STM에서 영감을 받은 optimistic parallel execution을 적용한다.

**동작 원리**: 스케줄러가 전달한 2δ 블록 내의 트랜잭션들을 메인 데이터베이스(원장)가 아닌, 완전히 격리된 **다중 버전 섀도우 메모리(Multi-Version Shadow Memory)** 상에서 병렬로 실행한다. 트랜잭션 간의 읽기/쓰기 충돌을 메모리 레벨에서 감지하고, 충돌이 발생한 트랜잭션만 재실행한다. 이 재실행은 섀도우 메모리 내부에서 포인터만 재조정하는 마이크로 롤백이며, 메인 원장에는 영향을 주지 않는다.

**3δ 확정 시 동작**: 합의의 글로벌 결과가 2δ 로컬 판단과 일치하면, 섀도우 메모리의 상태 변화량(State Diff)을 메인 원장에 병합한다. 불일치하면 해당 라운드의 섀도우 메모리 전체를 즉시 폐기(Drop)한다. 개별 트랜잭션을 추적할 필요 없이 메모리 블록을 버리면 되므로, 롤백 오버헤드가 사실상 제로이다.

**EVM 호환성**: REVM을 직접 수정하므로 EVM 바이트코드 호환을 유지한다. 기존 Solidity/Vyper 컨트랙트를 수정 없이 배포할 수 있다.

**Block-STM과의 차이**: Aptos의 Block-STM은 Move VM 위에서 동작하며, Move의 리소스 모델이 충돌 감지를 용이하게 한다. 본 프로젝트는 EVM의 계정/스토리지 모델 위에서 동작하므로, storage slot 단위로 read/write set을 추적하여 충돌을 감지한다. 합의가 결정한 순서에 따라 트랜잭션이 실행 파이프라인에 들어오므로, 순서 결정과 병렬 실행의 책임이 명확히 분리된다.

**효과**: 상태 충돌이 없는 트랜잭션(전체의 대다수)은 코어 수에 비례하여 병렬 처리되며, 단일 스레드 대비 수 배~수십 배의 처리량 향상이 가능하다.

### 3.3 Dual Optimistic의 결합 — 3-Stage Gate Pipeline

두 레이어가 결합되면 명확한 게이트 파이프라인이 형성된다.

```
[1δ: 순서 제안] → [2δ: 순서 잠정 확정 → 병렬 실행 개시] → [3δ: 최종 확정 → 원장 커밋]
                   ↑ Optimistic Consensus                    ↑ Gate: 여기서야 확정
                   ↑ Optimistic Execution (병렬)
```

합의는 순서만 결정하고, 2δ에서 그 순서를 실행 파이프라인에 넘긴다. 실행은 Block-STM으로 병렬 처리된다. 3δ 신호가 오면 실행 결과를 원장에 커밋한다.

**정상 경로(대다수)**: 2δ 순서가 3δ에서 확정되므로, 실행 결과가 그대로 원장에 내려간다. 합의 완료 시점에 실행이 이미 완료되어 있다.

**비정상 경로(극히 드묾)**: 비잔틴 equivocation으로 3δ에서 순서가 바뀌면, 실행 큐의 뭉치를 재정렬하고 재실행한다. 원장에는 아무것도 내려가지 않은 상태이므로 안전하다.

양쪽 모두 "잘못되면 되돌린다"는 동일한 철학이며, 되돌리는 비용이 모두 낮다(실행 큐 재정렬 / 트랜잭션 재실행). 복구 경로가 일관적이고 경량이다.

### 3.4 성능 모델 — 왜 기존 고성능 체인보다 빠른가

Δ를 노드 간 1홉 전파 지연, E를 1개 블록의 트랜잭션 배치 실행 시간이라 할 때:

**기존 모델** (3δ 확정 후 실행): 체감 완료 시간 = `3Δ + E`

**본 모델** (2δ 낙관적 병렬 실행): 체감 완료 시간 = `max(3Δ, 2Δ + E)`

**지연 단축 이득**: 두 모델의 차이는 항상 `min(Δ, E)`이며, 이는 시스템 환경에 관계없이 절대적으로 보장된다.

실행이 빠른 경우(E ≤ Δ)에는 실행 시간이 합의 대기 시간 내에 완전히 은닉되어, 체감 지연이 순수 합의 속도 `3Δ`로 수렴한다. 실행이 느린 경우(E > Δ)에는 기존 모델에서 유휴 상태로 버려지던 `1Δ`를 선행 연산에 활용하여 정확히 `Δ`만큼을 확정적으로 절약한다.

**충돌 저항성**: 트랜잭션 충돌로 재실행 시간 E_retry가 증가해도, 총 연산 시간 E = E_base + E_retry에 대해 동일한 수식이 적용된다. 기존 모델은 `3Δ + E_base + E_retry`, 본 모델은 `max(3Δ, 2Δ + E_base + E_retry)`. 충돌률이 폭증하더라도 본 모델의 `min(Δ, E)` 이득은 절대 삭감되지 않는다.

**배압 제어(Backpressure)**: 극단적으로 E가 합의 주기를 초과하면 실행 큐가 적체된다. 이를 방지하기 위해 스케줄러가 합의 엔진에 블록 제안 속도를 억제하는 피드백 루프를 내장한다.

### 3.5 HotStuff 계열 vs DAG — 낙관적 실행 결합 시 구조적 차이

동일한 2δ optimistic execution을 HotStuff 파생(MonadBFT, AptosBFT)과 DAG(Mysticeti)에 각각 결합했을 때:

**HotStuff + Optimistic Execution**: 단일 리더 구조. 리더가 비잔틴이거나 타임아웃 시 뷰 체인지가 발생하고, 선제 실행한 결과를 100% 폐기해야 하는 매크로 롤백이 발생한다. 새 리더가 블록을 전파할 때까지 파이프라인 전체가 정지하며, 네트워크 지연 페널티가 발생한다.

**DAG(Mysticeti) + Optimistic Execution**: 다중 제안자 구조. 비잔틴 노드가 존재해도 2δ 쿼럼 데이터가 DAG에 보존된다. 앵커 스킵 시 글로벌 순서만 재조정되며, 데이터 유실이 없다. 재연산은 섀도우 메모리 내부에서 CPU 스레드만 사용하는 마이크로 롤백이고, 합의 엔진은 멈추지 않는다.

핵심 차이는 **실패 모드의 비용**이다. HotStuff는 네트워크 페널티(파이프라인 정지), DAG는 CPU 페널티(국지적 재연산). 허가형 환경에서 비잔틴 노드 확률이 0에 수렴하므로 실질적 차이는 작지만, 아키텍처 수준에서 DAG가 낙관적 실행과의 결합에 구조적으로 더 적합하다.

## 4. Explicit Non-Goals (하지 않는 것)

명확하게 하지 않는 것을 정의한다.

**합의를 새로 발명하지 않는다.** Mysticeti는 SUI에서 프로덕션 검증된 DAG 기반 BFT 엔진이다. 본 프로젝트는 Mysticeti의 합의(순서 결정) 기능만을 추출하여 사용하며, SUI의 오브젝트 모델이나 Move VM과는 무관하다. 합의 엔진을 가져와서 EVM 실행 파이프라인과 결합하는 것이 본 프로젝트의 접근이다.

**범용 탈중앙화 L1을 지향하지 않는다.** 허가형(permissioned) 네트워크로 시작하며, 밸리데이터 셋을 통제한다. 이는 Mysticeti 2δ optimistic execution의 안전 가정을 강화하고, 성능을 예측 가능하게 만든다. 탈중앙화 확장은 성능과 안정성이 검증된 후 단계적으로 검토한다.

**자체 DA/파일 시스템/미디어 스트리밍을 구축하지 않는다.** Data Availability는 정산 대상 L1 또는 외부 DA 레이어에 위임한다. 핵심 가치는 합의와 실행의 최적화에 있다.

**자체 토큰 경제로 저장 인센티브를 설계하지 않는다.** 허가형 네트워크의 운영 비용은 운영 주체가 부담하며, 토큰 기반 인센티브는 필요 시 향후 설계한다.

## 5. Settlement & Security

### 5.1 허가형 모델의 보안

밸리데이터 셋이 통제되므로, BFT 안전 가정(f < n/3)이 운영적으로 강하게 유지된다. 이는 2δ optimistic execution에서 순서가 뒤집힐 확률을 사실상 0에 가깝게 만든다.

설령 비잔틴 equivocation이 발생하더라도, 3δ 게이트가 원장 커밋을 보호한다. 실행 결과는 3δ 신호 없이는 원장에 내려가지 않으므로, 합의 수준의 안전성이 실행/원장까지 관통한다.

밸리데이터 간 통신은 Mysticeti의 인증된 DAG 구조를 통해 이루어지며, 이중 제안(equivocation)은 DAG에서 즉시 감지되어 슬래싱 대상이 된다.

### 5.2 L1 정산

실행 결과(state root)는 정산 대상 L1에 주기적으로 앵커링된다. 정산 L1의 선택은 운영 요구사항(finality 모델, 수수료, 생태계)에 따르며, 본 설계는 특정 L1에 종속되지 않는다. 다만, EVM 호환 L1(Ethereum 등)이 자연스러운 정산 대상이다.

## 6. Roadmap

**Phase 0 — Dual Optimistic 검증**: Mysticeti DAG 합의의 순서 결정 출력을 EVM 실행 파이프라인에 연결. REVM + Block-STM 병렬 실행 엔진 구현. 2δ → 실행 → 3δ 커밋의 게이트 파이프라인 검증. 비잔틴 equivocation 시 실행 큐 재정렬 테스트. 허가형 테스트넷 운영.

**Phase 1 — 프로덕션 안정화**: 롤백 경로 검증(합의 롤백 + 실행 재실행). L1 정산 파이프라인 구현. 실 워크로드(DeFi, 토큰 전송) 기반 성능 측정 및 최적화.

**Phase 2 — 원장/상태 최적화**: 상태 트리 최적화(Merkle Patricia Trie 대안 검토). 상태 프루닝/아카이빙. 처리량 한계 확장.

**Phase 3 (선택) — 점진적 개방**: 밸리데이터 셋 확장 조건 정의. 개방 시 2δ optimistic execution의 안전 가정 재평가. 필요 시 3δ fallback 모드 도입.

---

## 7. Summary

본 프로젝트는 한 가지를 한다. **합의와 실행 양쪽에서 동시에 optimistic하게 동작하여, EVM 호환 트랜잭션의 확정 지연을 최소화하고 처리량을 극대화하는 것.** 합의는 검증된 엔진(Mysticeti)의 순서 결정 기능만 가져오고, 실행은 검증된 접근(Block-STM)을 격리된 섀도우 메모리 위에서 EVM에 적용하며, 3δ 게이트가 원장 안전을 보장한다. `min(Δ, E)`의 절대적 지연 단축이 수학적으로 보장되고, HotStuff 계열이 매크로 롤백(파이프라인 정지)을 감수해야 하는 자리에서 DAG 기반 합의는 마이크로 롤백(국지적 재연산)만 발생시킨다. 이것이 Dual Optimistic의 고유한 가치이다. 나머지는 하지 않는다.
